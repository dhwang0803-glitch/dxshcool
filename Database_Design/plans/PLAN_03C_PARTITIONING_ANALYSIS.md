## 1. 현황 요약

### 해결 대상: P03 (날짜 범위 조회)

```sql
-- P03 쿼리
SELECT COUNT(*), COUNT(DISTINCT user_id_fk), AVG(completion_rate)
FROM watch_history
WHERE strt_dt BETWEEN '2023-01-01' AND '2023-01-07 23:59:59';
```

| 항목 | 수치 |
|------|------|
| 대상 행 수 | 787,076건 (전체 19.7%) |
| 현재 성능 (cold) | 32,705ms |
| 현재 성능 (warm) | 15,315ms |
| **목표** | **<500ms** |
| 실행 계획 | Bitmap Index Scan → Bitmap Heap Scan |
| 접근 heap block 수 | **57,112 블록 (약 444MB)** |

### 왜 인덱스가 있는데 느린가

`idx_wh_strt_dt` 인덱스로 787K개의 행 위치를 찾는 것까지는 빠르다.
문제는 **heap block 접근** 단계다.

```
[인덱스] strt_dt 범위 → 787K 행 포인터 획득
    ↓
[heap] 37K 행 포인터 → 57,112개 heap block 무작위 접근 (444MB)
```

787K 행이 637MB 테이블 전체에 **흩어져 있기** 때문에, 실제로 읽어야 하는 block이
이론치(~19,600 블록)보다 약 **3배** 많은 57,112 블록이다.
파티셔닝의 핵심 효과는 이 **물리적 scatter를 제거**하는 것이다.

## 2. 파티셔닝 작동 원리 (PostgreSQL Partition Pruning)

```
[파티셔닝 전]
watch_history (3.99M rows, 637MB)
  └─ 모든 날짜 데이터가 혼재 → P03 범위의 787K rows가 57,112 블록에 산재

[파티셔닝 후 - 주별 예시]
watch_history (파티션 테이블)
  ├─ watch_history_w01 (2023-01-01~07, ~787K rows, ~145MB)  ← P03는 이것만 스캔
  ├─ watch_history_w02 (2023-01-08~14, ~900K rows, ~158MB)
  ├─ watch_history_w03 (2023-01-15~21, ~870K rows, ~153MB)
  ├─ watch_history_w04 (2023-01-22~28, ~900K rows, ~158MB)
  └─ watch_history_w05 (2023-01-29~31, ~540K rows, ~95MB)
```

**Partition Pruning**: WHERE strt_dt BETWEEN '2023-01-01' AND '2023-01-07'를 만족하는
파티션만 스캔 → 나머지 파티션은 **완전히 건너뜀**

## 3. 전략 옵션 비교

### 옵션 A: 월별 파티셔닝 (기존 계획)

```sql
PARTITION BY RANGE (strt_dt)
-- watch_history_2023_01: 2023-01-01 ~ 2023-02-01 (3.99M rows, 637MB)
-- watch_history_2023_02: 2023-02-01 ~ 2023-03-01
-- ...
```

| 항목 | 수치 |
|------|------|
| 파티션 수 (년간) | 12개 |
| 현재 데이터 파티션 수 | **1개** (2023-01 단일 월) |
| P03 스캔 대상 | **1개 파티션 전체 (637MB)** = 현재와 동일 |
| 단기 효과 | **없음** |
| 다월 데이터 시 효과 | Jan 1-7 쿼리가 Jan 파티션(약 100~200MB) 1개만 스캔 |

> **핵심 문제**: 현재 데이터가 2023-01 단일 월에 집중되어 있어
> 월별 파티셔닝은 **즉각적인 P03 개선 효과가 없다**.
> 다월 데이터가 쌓인 후에야 효과가 나타난다.

### 옵션 B: 주별 파티셔닝

```sql
PARTITION BY RANGE (strt_dt)
-- watch_history_2023_w01: 2023-01-01 ~ 2023-01-08 (~787K rows, ~145MB)
-- watch_history_2023_w02: 2023-01-08 ~ 2023-01-15
-- watch_history_2023_w03: 2023-01-15 ~ 2023-01-22
-- watch_history_2023_w04: 2023-01-22 ~ 2023-01-29
-- watch_history_2023_w05: 2023-01-29 ~ 2023-02-01
```

| 항목 | 수치 |
|------|------|
| 파티션 수 (년간) | 52개 |
| 현재 데이터 파티션 수 | **5개** |
| P03 스캔 대상 | **1개 파티션 (145MB)** |
| P03 heap block 수 | 57,112 → **~19,600 블록** (-66%) |
| 단기 효과 | **즉시, 현재 데이터만으로 P03 개선** |

### 옵션 C: CLUSTER + 인덱스 (파티셔닝 없이)

```sql
-- strt_dt 기준으로 테이블 물리적 재정렬
CLUSTER watch_history USING idx_wh_strt_dt;
```

| 항목 | 수치 |
|------|------|
| 구현 복잡도 | 낮음 (파티셔닝 DDL 불필요) |
| P03 heap block 감소 | 57,112 → ~19,600 블록 (-66%, 주별 파티션과 동등) |
| 단점 1 | **배타적 테이블 락 필요** — 작업 중 서비스 중단 |
| 단점 2 | INSERT/UPDATE 누적 시 정렬 깨짐 → 주기적 재실행 필요 |
| 단점 3 | 향후 파티셔닝 확장 시 재작업 |

## 4. P03 예상 성능: 주별 파티셔닝 기준

### I/O 감소 계산

```
현재: Bitmap Heap Scan 57,112 블록 (444MB, 무작위 접근)
주별: Seq Scan 19,600 블록 (153MB, 순차 접근)

블록 수 감소: 57,112 → 19,600 = 66% 감소
순차 vs 무작위 I/O 효과: 추가 3~5배 속도 향상 (SSD 환경)
종합 I/O 개선 배율: 5~10배
```

### 예상 성능 시나리오

| 시나리오 | Cold | Warm | 목표 달성 |
|---------|------|------|---------|
| 현재 (파티셔닝 없음) | 32,705ms | 15,315ms | ❌ |
| **주별 파티셔닝** | **~4,000~8,000ms** | **~1,500~3,000ms** | ❌ (단, 5~10배 개선) |
| 주별 파티셔닝 + 일별 집계 MV | ~5ms | ~0.1ms | ✅ |
| 월별 파티셔닝 (현재 단일월) | 32,705ms (동일) | 15,315ms (동일) | ❌ |

> **중요**: 파티셔닝만으로는 <500ms 목표 달성이 어렵다.
> **787K 행의 집계 연산(COUNT, AVG) 자체**가 VPC 1코어 환경에서 병목이기 때문이다.
> 완전한 해결은 **일별 집계 Materialized View** 추가가 필요하다.

## 5. 쿼리별 파티셔닝 효과 정리

### P02: VOD별 시청 통계 (1,714ms warm)

```sql
WHERE vod_id_fk = 'cjc|M5068430LFOL10619301'
-- 날짜 필터 없음
```

| 전략 | 효과 |
|------|------|
| 월별/주별 파티셔닝 | **없음** — strt_dt 기준 파티션이므로 vod_id 쿼리는 전체 파티션 스캔 |
| 비고 | 모든 파티션을 병렬 스캔 → 오히려 약간 느려질 수 있음 |

> P02는 `vod_id_fk` 기준 파티셔닝이 아니면 파티셔닝 효과 없음.
> 현재 warm 1,714ms는 목표(100ms) 초과이나, 이는 **최다 시청 VOD(71K회)** 케이스.
> 일반 VOD(수백~수천회)는 인덱스만으로 충분히 빠름.

### P03: 날짜 범위 1주 집계 (15,315ms warm)

```sql
WHERE strt_dt BETWEEN '2023-01-01' AND '2023-01-07'
```

| 전략 | Cold 예상 | Warm 예상 |
|------|----------|----------|
| 월별 파티셔닝 (현재 단일월) | 32,705ms (변화 없음) | 15,315ms (변화 없음) |
| **주별 파티셔닝** | **~5,000~8,000ms** | **~1,500~3,000ms** |
| 주별 파티셔닝 + 일별 집계 MV | ~5ms | ~0.1ms |

### P05: 사용자+날짜 복합 조회 (9~28ms)

```sql
WHERE user_id_fk = '...' AND strt_dt >= '2023-01-01'
```

| 전략 | 효과 |
|------|------|
| 주별 파티셔닝 | user_id_fk가 선두 컬럼 → 현행 커버링 인덱스가 더 효율적 |
| 비고 | 이미 PASS 상태 — 파티셔닝 영향 중립 (1~2ms 오차 범위) |

| 항목 | 내용 |
|------|------|
| DDL 복잡도 | 높음 |
| 파티션 수 (2023) | ~53개 |
| 향후 관리 | **주 1회 파티션 추가** — 자동화 스크립트 필요 |
| 다운타임 | 데이터 재적재 필요 (~30~60분) |
| 인덱스 재생성 | 파티션별 local index (~3~5시간 예상) |
| 주의사항 | 52 파티션 관리 복잡도 — pg_cron 또는 별도 관리 스크립트 필수 |

### 공통 작업 (월별/주별 모두)

```
Step 1: 파티션 테이블 생성 (watch_history_partitioned)
Step 2: 기존 watch_history → 새 테이블 COPY (30~60분)
Step 3: 파티션별 인덱스 재생성 (1~5시간)
Step 4: 테이블명 교체 (원자적 트랜잭션)
Step 5: 검증 후 구 테이블 삭제
```

## 7. 주별 파티셔닝 추가 구현 방안: 일별 집계 MV

P03의 완전한 해결(< 500ms)을 위한 보완책:

```sql
-- 일별 집계 Materialized View
CREATE MATERIALIZED VIEW mv_daily_watch_stats AS
SELECT
    DATE(strt_dt)                   AS watch_date,
    COUNT(*)                        AS total_views,
    COUNT(DISTINCT user_id_fk)      AS daily_active_users,
    AVG(completion_rate)            AS avg_completion
FROM watch_history
GROUP BY DATE(strt_dt);

CREATE UNIQUE INDEX ON mv_daily_watch_stats (watch_date);

-- P03 대체 쿼리 (1주 → 7개 행 집계)
SELECT
    SUM(total_views),
    SUM(daily_active_users),   -- 주의: DISTINCT 사용자 수는 중복 포함 근사치
    AVG(avg_completion)
FROM mv_daily_watch_stats
WHERE watch_date BETWEEN '2023-01-01' AND '2023-01-07';
-- 예상: <10ms
```

> 단, `COUNT(DISTINCT user_id_fk)` 주 단위 정확한 집계는
> MV에서 근사치만 가능 (일별 중복 사용자 카운팅 불가피).
> 정확도가 중요한 경우 허용 오차 협의 필요.

## 8. 권고안 요약

| 구분 | 권고 | 이유 |
|------|------|------|
| **1순위** | 주별 파티셔닝 | 현재 단일월 데이터에서도 P03 즉각 개선 |
| **2순위** | + 일별 집계 MV | P03 <500ms 완전 달성 |
| **3순위** | 월별 파티셔닝 | 관리 편의성은 높으나 단기 효과 없음 |
| **제외** | CLUSTER 단독 | 배타적 락 필요, 장기 지속성 없음 |

### 팀 협의 결정 포인트

1. **P03 타겟**: "파티셔닝만으로 근접(~2,000ms)"으로도 충분한지, 아니면 MV로 <500ms 완전 달성이 필요한지
2. **파티션 관리 부담**: 주별 52개 파티션 자동화 스크립트를 누가 관리할지
3. **다운타임 일정**: 데이터 재적재 (~1~2시간) 일정 조율 (심야 배치)
4. **P02 전략**: 파티셔닝과 별개로 P02 개선이 필요한지 — VOD별 집계 MV 추가 검토
