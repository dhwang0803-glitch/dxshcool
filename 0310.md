[shm.dll 버그 원인]

Windows에서 DLL 로딩 순서 문제.

torch를 직접 import하면
→ Python이 conda 환경 PATH에서 필요한 DLL들을 정상적으로 찾아서 로드함.

sentence_transformers → transformers → torch 순으로 경유해서 import하면
→ transformers가 먼저 로드되는 과정에서 Windows DLL 검색 경로가 달라져서
  shm.dll의 의존 DLL을 못 찾음.

쉽게 말하면:
- torch 혼자 import: "나 직접 들어갈게" → 경로 정상
- transformers 통해서 import: "transformers가 먼저 들어가서 환경 바꿔놓음"
  → torch가 들어오려니 DLL 경로가 어긋남

왜 이 환경에서만?
- Windows + conda 환경 특유의 DLL 격리 문제
- Linux/Mac은 이런 문제 없음
- conda 환경의 Library/bin에 있는 DLL들이 특정 import 순서에서 PATH에 없는 것처럼 동작함

해결: batch_embed.py 상단에 import torch 선행 추가 (sentence_transformers보다 먼저)


[CF_Engine 폴더 구조]

src/
  import 전용 라이브러리. 직접 실행 X.
  - data_loader.py  : watch_history DB → csr_matrix 변환, 인코더/디코더 반환
  - als_model.py    : ALS 학습 + 전체 유저 배치 추천 생성
  - recommender.py  : 인덱스 → vod_id_fk 역변환, 레코드 포매팅

scripts/
  직접 실행하는 진입점. python scripts/xxx.py 로 실행.
  - train.py        : 전체 파이프라인 실행 (--dry-run 지원)
  - export_to_db.py : serving.vod_recommendation 테이블에 결과 적재
  - evaluate.py     : NDCG@K / MRR / HitRate@K 성능 평가
  - pilot_test.py   : 파일럿 테스트 (ALS 속도 측정, poster_url 커버리지 확인)

tests/
  pytest 단위 테스트. DB 없이 로컬에서 실행 가능. 9/9 PASSED.
  - test_data_loader.py  : 희소 행렬 shape, 인코더 정합성, confidence 값
  - test_als_model.py    : 학습 후 벡터 shape, 추천 결과 K개
  - test_recommender.py  : 역매핑 정확성, 출력 형식, recommendation_type

config/
  als_config.yaml — 하이퍼파라미터, recommendation_type 등 설정값
  (factors=128, iterations=20, alpha=40, top_k=20, recommendation_type="CF")

docs/
  설계 문서, 평가 리포트 저장 (eval_report_날짜.md)

plans/
  구현 계획서 (implementation_plan.md) — 구현 현황 반영
