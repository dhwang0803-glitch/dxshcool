## 5. 파일럿 중 발견된 버그 및 수정 내역

> **참고 — 버그 1 파일럿 한정 적용 배경**
>
> Full 운영에서는 시리즈의 각 에피소드마다 영상 내용이 유의미하게 달라지므로, 에피소드별로 개별 트레일러를 수집하고 각각 독립적인 벡터 임베딩을 생성해야 한다. 따라서 아래 버그 1의 `dedup_by_series` 처리(시리즈당 1개 대표 에피소드만 추출)는 파일럿의 100건 제한 조건에서만 적용한 전략이며, Full 운영 코드에서는 제거된다.

### 버그 1 — 층화 샘플에 동일 시리즈 에피소드 중복 포함 (파일럿 한정)

**현상**
TV드라마 40개 할당 중 "황제의 딸 1" 1화~40화가 연속으로 포함.

**원인**
`stratified_sample`이 `full_asset_id` 오름차순 정렬 후 앞에서 quota개를 단순 절단하는 방식으로 동작.

**수정 (파일럿 한정)**
`dedup_by_series()` 함수 추가. `SERIES_CT_CL`(TV드라마 / TV연예오락 / TV시사교양 / TV애니메이션)에서 `strip_episode_suffix` 기준 시리즈명이 같으면 가장 낮은 `full_asset_id` 1개만 유지하여 100건 샘플 내에서 다양한 시리즈를 확보.

---

### 버그 2 — 에피소드 suffix 제거 regex가 붙여쓰기 처리 못함

**현상**
"파리의연인01화" → `strip_episode_suffix`가 "파리의연인"으로 변환 실패 → dedup 미적용 → 파리의연인01화~14화 연속 샘플링.

**원인**
regex `\s+\d+\s*[회화]$`에서 `\s+`(공백 1개 이상)가 필수이므로 공백 없이 붙은 케이스를 처리하지 못함.

**수정**

```python
# 수정 전
\s+\d+\s*[회화]$

# 수정 후
\s*\d+\s*[회화]$
```

`\s+` → `\s*`로 변경하여 공백이 없는 붙여쓰기 케이스도 처리.

---

### 버그 3 — 유튜브 검색 쿼리: 숫자-한글 붙여쓰기로 검색 실패

**현상**
DB 저장 제목 "1724기방난동사건"으로 검색 시 유튜브에서 결과 없음. 수동 검색 시 "1724 기방난동사건 예고편"으로 검색하면 결과 확인 가능.

**원인**
DB 제목에 숫자-한글 사이 공백 없음.

**수정**
`normalize_title()` 함수 추가. 숫자↔한글 경계에 공백 삽입. 영화 카테고리 검색 쿼리에 우선 적용.

---

### 버그 4 — `duration_filter`가 모든 영상을 차단

**현상**
크롤링 첫 실행 시 전 건 `no_result` 실패.

**원인**
yt-dlp의 `match_filter`는 검색 결과 초기 단계(`incomplete=True`)에서도 호출되는데, 이때 `duration=None` → `0`으로 처리되어 "너무 짧음(0초)"으로 전부 탈락.

**수정**

```python
def duration_filter(info, *, incomplete):
    if incomplete:
        return None
    # 기존 필터 로직 ...
```

`duration_filter`에 `if incomplete: return None` 추가.

## 8. Full 운영 전 개선 필요 사항

### 8-1. 동일 유튜브 파일을 참조하는 복수 vod_id 처리

"타짜"와 "타짜 스페셜(감독판)", "타짜 소장판"처럼 제목 변형 콘텐츠는 유튜브에서 동일한 트레일러 영상이 검색된다. 이 경우 같은 파일이 여러 `vod_id`에 매핑되며, `--delete-after-embed` 옵션을 사용하면 첫 번째 `vod_id` 임베딩 후 파일이 삭제되어 나머지 `vod_id`가 cascade 실패한다.

**개선 방향**
- `batch_embed.py`에서 임베딩 전 파일명 기준으로 `vod_id` 그룹을 구성하고, 파일당 1회만 임베딩 후 해당 벡터를 같은 파일을 참조하는 모든 `vod_id`에 복사 적재.
- 파일 삭제는 해당 그룹 전체 처리 완료 후 수행.

---

### 8-2. 유튜브 미등재 콘텐츠 fallback 처리

교육 콘텐츠("Feel n Speak" 등)처럼 유튜브에 등재되지 않은 콘텐츠는 크롤링 단계에서 `no_result`로 실패한다. 카테고리와 무관하게 유튜브 검색에 실패한 모든 콘텐츠에 fallback을 적용해야 한다.

**fallback 후보 전략 (우선순위 순)**

| 순위 | 전략 | 방식 | 비고 |
|------|------|------|------|
| 1 | CLIP 텍스트 임베딩 | `제목 + 장르 + ct_cl` 문자열을 CLIP text encoder로 인코딩 | 영상 벡터와 동일한 512차원 공간 공유. 추가 인프라 불필요 |
| 2 | KMDb 포스터 이미지 | 한국영화데이터베이스 API로 포스터 이미지 수집 후 이미지 임베딩 | 시각 정보 유지. API 키 필요 |

- fallback 적용 여부는 `crawl_status.json`의 `embed_mode: "video" | "text" | "poster"` 필드로 기록하여 이후 품질 분석에 활용.
- fallback으로 생성된 벡터는 `vod_embedding` 테이블의 `model_name` 컬럼에 `"clip-ViT-B-32-text"` 등으로 구분 저장.

---

### 8-3. 크롤링 병렬화 전략

현재 단일 프로세스 기준 약 300시간 소요. 아래 전략을 단계적으로 적용한다.

#### 단기 — ct_cl별 프로세스 분리

기존 `--ct-cl` 인자를 활용해 카테고리별로 별도 프로세스를 실행한다. 각 프로세스는 독립적인 `crawl_status.json` 파티션(또는 별도 파일)에 체크포인트를 저장한다.

```bash
# 예시: 카테고리별 병렬 실행
python crawl_trailers.py --ct-cl TV드라마 &
python crawl_trailers.py --ct-cl 영화 &
python crawl_trailers.py --ct-cl "TV 연예/오락" &
python crawl_trailers.py --ct-cl TV애니메이션 TV시사/교양 다큐 &
wait
```

- 4개 워커 기준 약 75시간으로 단축 (300h / 4).
- 동일 IP에서 동시 요청이 늘어나므로 워커당 요청 간격을 2~3초로 유지.

#### 중기 — 요청 간격 최적화 + 429 적응형 대기

현재 고정 1.5~3초 대기를 아래와 같이 개선한다.

| 상황 | 대기 시간 |
|------|---------|
| 정상 응답 | 1.0~2.0초 |
| 429 첫 번째 | 60초 |
| 429 연속 2회 | 300초 |
| 429 연속 3회 이상 | 해당 워커 일시 중단 후 재개 |

#### 장기 — 시리즈 레벨 크롤 dedup (YouTube 요청 수 감소)

유튜브 크롤은 시리즈 단위로 1회만 수행하고, 다운로드된 트레일러 파일을 해당 시리즈의 모든 에피소드에 공유한다. 에피소드별 실제 임베딩은 스트리밍 플랫폼 원본 영상(VOD 파일)에서 직접 추출하는 방향으로 전환한다.

| 방식 | YouTube 요청 수 | 임베딩 입력 |
|------|---------------|-----------|
| 현행 (에피소드별 크롤) | 166,159건 | YouTube 트레일러 |
| 개선 (시리즈 크롤 + 원본 임베딩) | 고유 시리즈 수 (추정 수천~수만) | 실제 VOD 파일 |

이 방식으로 전환하면 YouTube 크롤 요청이 대폭 감소하고, 에피소드별로 의미있는 고유 벡터를 확보할 수 있다.
