## 2. 주요 결정: 병렬화 vs 딜레이 축소

### 검토 배경

1차 수집 속도가 ~1.3건/분으로 전체 완료까지 약 58시간(~2.4일) 예상되어 속도 개선 방안 검토.

### 병렬화 검토 결과 (비채택)

| 항목 | 판단 |
|------|------|
| JSON 충돌 | Lock으로 해결 가능 |
| YouTube 429 리스크 | workers 증가 시 요청량 비례 증가 → 429 빈도 증가 |
| 속도 개선 | 선형 비례 아님 (요청-대기-응답 병목, CPU 병목 아님) |
| 원인 추적 | 문제 발생 시 단일 실행보다 복잡 |

**결론: 현재 본수집 단계에서는 병렬화 비채택.**
→ 2차 재시도(failed 전용) 단계에서 workers=2 소규모만 검토 예정.

### 딜레이 축소 (채택)

```python
# 변경 전
REQUEST_DELAY_MIN = 1.5
REQUEST_DELAY_MAX = 3.0

# 변경 후
REQUEST_DELAY_MIN = 0.5
REQUEST_DELAY_MAX = 1.0
```

- 코드 변경 최소 (2줄)
- 체크포인트 보존 (이어서 실행)
- 429 발생 시 기존 60초 대기 로직으로 자동 처리
- 예상 효과: ~1.3건/분 → ~2.5~3건/분 (약 2배)

## 4. FAIL 패턴 분석

### 4-1. YouTube 원본 부재 프로그램

- `위대한탄생 시즌2`: 다수 회차 no_result (2~4회차 등) — YouTube 업로드 자체 없음
- `나는가수다2`: 다수 회차 no_result — MBCkpop 영어 제목으로만 업로드

### 4-2. KBS 날짜형식 업로드 패턴 (2차 과제)

KBS 공식 채널이 에피소드 번호 대신 방송 날짜(YYMMDD)를 제목에 사용:

- DB: `우리동네 예체능 04회`
- YouTube 실제 제목: `"130409 첫 방송 우리동네예체능 예고 | KBS방송"`

1차 파이프라인에서 failed 처리되는 이유:
1. yt-dlp 날짜쿼리 (`우리동네 예체능 130409`) → no_result
2. 설령 검색에 걸려도 ep_int 필터 (`4회`, `E04` 없음) → 차단

→ **2차 재시도 때 air_date→YYMMDD 변환 + ep_int 완화로 대응 예정**

### 4-3. duration 초과 케이스

- `꽃보다 할배 01회` 등: 검색 결과 duration=[132, 613, 329] → 30~600초 내 후보 있으나 title_matches 불통과
