## 3. WebSocket `/ad/popup` 프로토콜

| 상태 | 항목 |
|------|------|
| 🟢 | **WebSocket (양방향)** 확정 — 팝업 축소/확대/제거 인터랙션 필요 |
| 🟢 | 메시지 payload — 아래 2가지 타입별 구조 확정 |
| 🟢 | 광고 액션 — **시청예약만** (채널이동 제거). 같은 WS 채널로 응답 |
| 🟢 | 광고 타입: `local_gov`(지자체) / `seasonal_market`(제철장터) |

**광고 팝업 동작 규칙 (2026-03-20 확정):**

**타입 1: 지역광고 (`local_gov`)**
- 지역 사진 + 축제 이름 + 축제 일정이 포함된 움직이는 광고 팝업 출력
- 유저 선택 버튼 없음
- **10초 후 자동 최소화** → 파란색(다시 열기) / 빨간색(완전 제거) 작은 버튼만 남김

**타입 2: 제철장터 (`seasonal_market`)**
- "채널 25번 제철장터에서 현재 {사물인식된 상품명}이 판매중입니다" 메시지
- 버튼: **시청예약하기** / **종료하기**
- 시청예약 클릭 → "시청예약되었습니다" 팝업 2초 후 자동 제거
- 종료하기 클릭 → 즉시 팝업 제거
- **10초 후 자동 최소화** (미응답 시) → 파란색/빨간색 버튼

**Server → Client payload:**
```json
{
  "type": "ad_popup",
  "ad_type": "local_gov",
  "vod_id": "string",
  "time_sec": 120,
  "data": {
    "image_url": "string",
    "festival_name": "보령머드축제",
    "festival_schedule": "2026-07-18 ~ 2026-07-27"
  }
}
```
```json
{
  "type": "ad_popup",
  "ad_type": "seasonal_market",
  "vod_id": "string",
  "time_sec": 120,
  "data": {
    "product_name": "제주 감귤",
    "channel": 25,
    "broadcast_time": "14:00"
  }
}
```

**Client → Server 액션:**
```json
{ "type": "ad_action", "action": "reserve_watch", "vod_id": "string" }
{ "type": "ad_action", "action": "dismiss", "vod_id": "string" }
{ "type": "ad_action", "action": "minimize", "vod_id": "string" }
{ "type": "ad_action", "action": "reopen", "vod_id": "string" }
```

## 4. JWT 인증

| 상태 | 항목 |
|------|------|
| 🟢 | `POST /auth/token` — `{ "user_id": "sha2_hash" }` 전송, 셋톱박스 자동 로그인 |
| 🟢 | JWT 저장 — **불필요** (셋톱박스 켜면 자동 인증, 만료 없음) |
| 🟢 | 토큰 만료 — **없음** (셋톱박스 전원 = 인증 세션) |
| 🟢 | 비로그인 유저 — **고려 불필요** (사이트 진입 = 즉시 로그인) |

**인증 방식 (2026-03-20 확정):**
- IPTV 셋톱박스에 유저 ID(sha2_hash)가 등록되어 있음
- 셋톱박스 전원 ON → 자동으로 `/auth/token` 호출 → JWT 발급 (만료 없음)
- 별도의 비밀번호, OAuth, refresh token 불필요
- 모든 엔드포인트는 인증 필수 (비로그인 시나리오 없음)

## 5. Database_Design 브랜치 협의 — 신규 테이블 4개

> 현재 스키마에 없는 테이블. `Database_Design` 브랜치에 DDL 추가 요청 필요.

### 5-1. `public.wishlist`

```sql
CREATE TABLE public.wishlist (
  user_id     VARCHAR(64)  NOT NULL,
  series_id   VARCHAR(64)  NOT NULL,
  created_at  TIMESTAMPTZ  DEFAULT now(),
  PRIMARY KEY (user_id, series_id)
);
```

- 찜 추가/해제 (시리즈 상세 + 마이페이지)
- CF_Engine 피드백 신호 활용 검토 필요 (찜 = 암묵적 긍정, watch_history보다 약한 가중치 0.5)

### 5-2. `public.episode_progress`

```sql
CREATE TABLE public.episode_progress (
  user_id         VARCHAR(64)  NOT NULL,
  episode_id      VARCHAR(128) NOT NULL,
  series_id       VARCHAR(64)  NOT NULL,
  completion_rate SMALLINT     NOT NULL DEFAULT 0 CHECK (completion_rate BETWEEN 0 AND 100),
  watched_at      TIMESTAMPTZ  DEFAULT now(),
  PRIMARY KEY (user_id, episode_id)
);
CREATE INDEX idx_ep_progress_series ON public.episode_progress (user_id, series_id, watched_at DESC);
```

- 에피소드별 시청 진행률 → "1화 시청하기" vs "이어보기" 분기
- 기존 `watch_history`(시리즈 단위)와 역할 분리: `watch_history`는 홈/마이페이지, `episode_progress`는 시리즈 상세/플레이어
- `completion_rate` 업데이트 주기: **30초 heartbeat** 확정 (2026-03-21)

### 5-3. `public.purchase_history`

```sql
CREATE TABLE public.purchase_history (
  purchase_id   SERIAL       PRIMARY KEY,
  user_id       VARCHAR(64)  NOT NULL,
  series_id     VARCHAR(64)  NOT NULL,
  option_type   VARCHAR(16)  NOT NULL CHECK (option_type IN ('rental', 'permanent')),
  price         INTEGER      NOT NULL,
  purchased_at  TIMESTAMPTZ  DEFAULT now(),
  expires_at    TIMESTAMPTZ,
  UNIQUE (user_id, series_id, purchased_at)
);
```

- 구매 기록 + 대여(48h) 만료 관리
- `expires_at > now()` 조건으로 대여 만료 체크

### 5-4. `public.point_history`

```sql
CREATE TABLE public.point_history (
  history_id   SERIAL       PRIMARY KEY,
  user_id      VARCHAR(64)  NOT NULL,
  type         VARCHAR(8)   NOT NULL CHECK (type IN ('use', 'earn')),
  amount       INTEGER      NOT NULL,
  description  VARCHAR(256) NOT NULL,
  created_at   TIMESTAMPTZ  DEFAULT now()
);
```

- 포인트 차감/적립 내역
- `POST /purchases` 시 purchase_history와 트랜잭션으로 묶어 INSERT

### 8-1. `series_id` 컬럼 부재 — 🟢 결정 완료

Frontend는 모든 곳에서 `series_id`를 식별자로 사용하지만, `vod` 테이블에 `series_id` 컬럼이 없었다.

| DB 실제 | Frontend 기대 |
|---------|--------------|
| `full_asset_id` VARCHAR(64) — PK, 에피소드 단위 | `series_id` — 시리즈 단위 |
| `series_nm` VARCHAR(255) — 시리즈 이름(텍스트) | 시리즈 식별용 ID |

**결정 (2026-03-20)**: `series_nm`을 식별자로 사용한다.
- 시즌 정보가 포함된 VOD의 경우, `series_nm`에 시즌 정보를 붙여서 구분 (예: `"눈물의 여왕 시즌1"`)
- Frontend에 `series_id` 대신 `series_nm`으로 식별하도록 전달
- `series_nm`은 시리즈 그룹핑 키 (15,090 시리즈 / 169,581 VOD) → UNIQUE가 아닌 **일반 인덱스**
- 🟢 **인덱스 추가 완료 (2026-03-20)**: API 조회 패턴에 최적화된 커버링 인덱스 적용
  ```sql
  CREATE INDEX idx_vod_series_nm_cover
      ON public.vod (series_nm)
      INCLUDE (full_asset_id, asset_nm, ct_cl, poster_url);
  ```
  기존 `idx_vod_series_nm` (단순 B-tree)에 추가로, 시리즈 상세/홈 섹션 조회 시 인덱스만으로 응답 가능

**`series_nm` 기반 에피소드 조회 시 실 데이터 이슈 (2026-03-20 확인)**:

1. **에피소드 중복**: 같은 에피소드가 다른 `full_asset_id`로 중복 존재 (예: "짝 08회" → `cjc|M4903632LSGH50604201`, `cjc|M4903632LSGL21170301` — 동일 provider, asset_id만 상이)
   - → API에서 `DISTINCT ON (asset_nm)` 또는 provider 우선순위로 중복 제거 필요
2. **시즌 미구분 시리즈**: "아는 형님"처럼 시즌 구분 없이 수백 회차가 하나의 `series_nm`에 포함
   - → 에피소드 목록 API에서 **페이지네이션 필수** (offset/limit 또는 cursor)
3. **시즌 구분 현황**: 일부만 `series_nm`에 시즌 접미사 존재 (예: `시계마을티키톡 시즌1`, `캐리앤하우스 시즌1`)
   - → 담당자 결정대로 시즌 정보 포함 VOD는 `series_nm` 갱신 작업 필요 (asset_nm에서 파싱)

### 8-4. `completion_rate` 타입 불일치 — 🟢 결정 완료

| 위치 | 타입 | 범위 |
|------|------|------|

**결정 (2026-03-20)**:
- `episode_progress` 테이블에 `completion_rate` SMALLINT(0~100) 컬럼을 만들어 **사전 연산 결과를 정수 단위로 저장**

- **API 응답 통일**: 모든 엔드포인트에서 `episode_progress.completion_rate` (정수 0~100)를 사용

- `watch_history`는 ML 파이프라인 전용 (추천 엔진 입력)으로 한정, API 응답에서는 사용하지 않음

**핵심 원칙 (2026-03-20 확정)**:

**1. API 응답에서 내부 ID값 일체 미노출 (MANDATORY)**
- `full_asset_id`, `sha2_hash`, `watch_history_id`, `vod_embedding_id` 등 DB 내부 식별자를 Frontend에 절대 전달하지 않는다
- 시리즈 식별: `series_nm` (이름 기반)
- 에피소드 식별: `asset_nm` (이름 기반)
- 유저 식별: JWT 토큰 기반 (내부적으로 sha2_hash 매핑, 외부 노출 X)
- **유일한 예외**: `user_name` = `sha2_hash` 앞 5자 파싱 (표시명 용도)

**2. 내부 정규화 데이터 미노출**
- `watch_history` raw 데이터를 유저에게 직접 노출하지 않는다
- 마이페이지 시청 내역(`GET /user/{id}/history`)은 `episode_progress` 데이터로 대체
- API 응답 필드는 유저가 이해 가능한 값만 포함 (이름, 포스터, 진행률, 날짜 등)

**3. 실시간 데이터 처리 (방안 A 확정 — 2026-03-21)**
- 인프라 제약(1GB RAM)으로 **Redis 미도입**. PG 내장 기능 + 인메모리 버퍼로 해결
- 시청 진행률: 인메모리 버퍼 → 60초 batch UPSERT
- 포인트 잔액: `user.point_balance` 캐시 컬럼 + DB 트리거 자동 갱신
- 마이페이지 실시간 갱신: PG LISTEN/NOTIFY → WebSocket push
- 시청예약 알림: 30초 주기 background task → WebSocket push
- 상세 설계: `API_Server/docs/realtime_architecture.md`

**4. FOD 무료 콘텐츠 UI 분기 (MANDATORY)**
- `vod.asset_prod = 'FOD'`인 시리즈는 **무료 콘텐츠** → 구매 불필요
- 시리즈 상세 페이지 진입 시 UI 분기:
  - FOD 시리즈: **"1화 시청하기"** 버튼 (구매 절차 없이 바로 재생)
  - 유료 시리즈 (RVOD/SVOD/기타): **"구매하기"** 버튼 → 구매 페이지 이동
- `GET /vod/{series_nm}` 응답에 `is_free: boolean` 필드 포함하여 Frontend에서 분기 처리
- `GET /vod/{series_nm}/purchase-options` — FOD 시리즈는 빈 배열 또는 `{ "is_free": true }` 반환
- purchase_history 초기 적재: FOD 제외 전부 구매 대상. legacy backfill은 `points_used=0`, `option_type='permanent'`
